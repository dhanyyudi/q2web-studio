# qgis2web Studio — Comprehensive Audit (2026-04-29)

> **Status:** Documentation only. No code has been changed by this audit. All action items below are intended for execution by Codex in subsequent PRs, organized by sprint priority.

## 1. Context

`qgis2web-studio` is a Vite + React 18 + Leaflet + Radix UI single-page editor that ingests qgis2web Leaflet exports (folder or ZIP), lets the user edit branding, layer style, popup, and legend, and emits a static ZIP with a custom Studio runtime. The MVP is functional, but a hands-on user review surfaced six concrete pain points and several open questions about UX, large-data robustness, and security posture.

This audit covers four dimensions:

1. **UI / UX clarity and information architecture**
2. **Functional gaps and editor ergonomics**
3. **Web and data security posture**
4. **Robustness with large GeoJSON datasets**

It is structured so that Codex can pick up each action item with full file/line context and a clear definition of done.

## 3. Findings — User Pain Points

### 3.1 Inspector mixes project-global and per-layer concerns (Pain Point 1)

**Code evidence**

- [src/App.tsx:480](../src/App.tsx#L480) through 599 places all five tabs (Branding, Layer, Style, Popup, Legend) inside a single `Tabs.Root` with no contextual filtering.
- The **Branding** tab holds project-global controls (map title, footer, logo, theme colors, header height) that are not relevant while the user is editing one layer's style.
- The **Style** tab exposes a flat form: `pointRadius`, `fillColor`, `strokeWidth`, `dashArray`, `categoryField`, etc. shown for any layer regardless of geometry.
- `selectedLayer.geometryType` is available ([src/types/project.ts:42](../src/types/project.ts#L42)) but never used to gate the UI.

**Recommendation**

1. **Structurally separate** *Project Settings* (Branding / Theme / Map) from *Layer Editor* (Layer / Style / Popup / Legend). Two inspector modes:
   - **Project mode** — no layer selected, or the user clicks a "Project" entry: only Branding + Map View + Theme.
   - **Layer mode** — a layer is highlighted: only Layer + Style + Popup + Legend. Branding is hidden.
2. **Geometry-aware Style tab**, driven by `selectedLayer.geometryType`:
   - `Point` / `MultiPoint` → show `pointRadius`, `fillColor`, `fillOpacity`, `strokeColor`, `strokeWidth`. Hide `dashArray` (except for the marker outline edge case).
   - `LineString` / `MultiLineString` → show `strokeColor`, `strokeWidth`, `strokeOpacity`, `dashArray`. Hide `fillColor`, `fillOpacity`, `pointRadius`.
   - `Polygon` / `MultiPolygon` → show `fillColor`, `fillOpacity`, `strokeColor`, `strokeWidth`, `dashArray`. Hide `pointRadius`.
3. **Inspector breadcrumb** at the top: `Project › Zona Nilai Tanah` so the user always knows which scope is being edited.
4. **Geometry-aware draw toolbar** ([App.tsx:422-445](../src/App.tsx#L422)): disable point/line/polygon draw buttons when the active layer's geometry type does not match (e.g. disable the point tool when a polygon layer is active).

### 3.2 Attribute table is not flexible (Pain Point 2)

**Code evidence**

- [src/styles.css:486](../src/styles.css#L486): `.table-scroll { height: 181px }` is hardcoded.
- [src/App.tsx:619](../src/App.tsx#L619): `slice(0, 80)` caps rendering at 80 rows.
- The main-stage grid `grid-template-rows: 46px minmax(360px, 1fr) 230px` ([styles.css:133-138](../src/styles.css#L133)) gives the table a static 230 px row.

**Recommendation**

1. **Vertical splitter** between the map and the attribute table using `react-resizable-panels` (small, headless, integrates cleanly with Tailwind / shadcn). Persist the most recent height to OPFS or `localStorage`.
2. **Header controls** on the attribute panel: minimize, maximize (overlay the map), and close.
3. **Virtualize** rows with `@tanstack/react-virtual`. Drop the 80-row cap. Replace it with an info bar ("Showing 1–100 of 5,234 features") plus a filter input.
4. **Resizable columns** via TanStack Table v8 column resizing.

### 3.3 Basemap blank (Pain Point 3) — CRITICAL ROOT CAUSE

**Root cause**

- [vite.config.ts:9](../vite.config.ts#L9): `Cross-Origin-Embedder-Policy: require-corp` is enabled for both dev server and preview.
- COEP `require-corp` requires every cross-origin subresource to advertise `Cross-Origin-Resource-Policy: cross-origin` or pass CORS. The third-party tile servers in use — `server.arcgisonline.com`, `tile.openstreetmap.org`, `basemaps.cartocdn.com` — do **not** send the CORP header, so the browser blocks every tile and the map appears blank.
- The COEP header was originally added for OPFS isolation, but OPFS does **not** require COEP — it only needs a secure context.

**Recommendation (in priority order)**

1. **Remove `Cross-Origin-Embedder-Policy: require-corp`** from `vite.config.ts`. OPFS continues to work without it. This is the immediate fix.
2. *(Optional, only if SharedArrayBuffer-based isolation is needed later.)* Switch to `Cross-Origin-Embedder-Policy: credentialless`, which is permissive for credential-less public subresources.
3. Add a **tile error overlay** in MapCanvas: subscribe to `tileLayer.on("tileerror", …)` and emit a toast like "Basemap failed to load: check your connection or pick a different basemap." Today these errors are silent.
4. Set `crossOrigin: "anonymous"` on the `L.tileLayer({...})` call ([src/components/MapCanvas.tsx:234-249](../src/components/MapCanvas.tsx#L234)) to explicitly opt into CORS — useful if a future canvas read-back or screenshot feature is added.
5. **Expand the basemap catalog** with options popular in Indonesia: Esri World Topo, Esri World Street, OSM HOT, Stamen Terrain (via Stadia Maps). Document the attribution and ToS for each.

### 3.4 Should the UI be redesigned with shadcn/ui + Tailwind? (Pain Point 4)

**Recommendation: yes, but in phases.**

Rationale

- The app already uses Radix UI primitives ([package.json](../package.json) lines 14-19). shadcn/ui is a composer pattern on top of the same Radix primitives, so the migration is non-invasive.
- The custom `src/styles.css` (~600+ lines) is starting to be expensive to maintain. Tailwind + shadcn would: (a) reduce bespoke CSS, (b) align with the token-based theming already implied by `theme.accent` and friends, (c) make a future dark mode straightforward, (d) ship a battle-tested component library (Resizable, DataTable, Sheet, Dialog) with mature a11y.
- Brand identity (teal `#156f7a`, `q2` mark) should be preserved by mapping it into `tailwind.config.ts` tokens.

Three-phase migration

1. **Phase 1 — Tooling.** Add Tailwind v4 (or v3.4 if v4 is not yet stable on Vite 6), the shadcn `@/components/ui` scaffold, `tailwind-merge`, `class-variance-authority`, `clsx`. Move theme tokens into `tailwind.config.ts`. Keep `styles.css` for Leaflet-specific overrides.
2. **Phase 2 — Shell refactor.** Replace the topbar, side panel, and inspector container with shadcn components (Sheet / Sidebar / Tabs / Card / Button). Use `react-resizable-panels` for the map↔table splitter.
3. **Phase 3 — Sub-component refactor.** Attribute table → shadcn DataTable + virtualized TanStack Table. Style editor → shadcn Form + react-hook-form + zod schema. Color picker → `react-colorful` inside a Radix Popover (today it is a bare `<input type="color">`).

### 3.5 Per-layer preview, legend placement, do all layers need a legend? (Pain Point 5)

**Code evidence**

- Legend in editor preview: [src/components/MapCanvas.tsx:222-227](../src/components/MapCanvas.tsx#L222), positioned at `position: absolute; right: 14px; bottom: 54px`.
- Legend in exported runtime: [src/runtime/runtime.ts:102-123](../src/runtime/runtime.ts#L102), positioned at `position: fixed; right: 14px; bottom: 52px`.
- The legend is a single global block that merges entries from every visible layer. There is no per-layer grouping and no link between the layer toggle and the legend section.

**Recommendation**

1. **Per-layer legend sections** inside the global legend, grouped by layer name. When a layer is toggled off via the layer control, its legend section is hidden too.
2. **Legend position picker** in the Branding tab: top-left, top-right, bottom-left, bottom-right (at minimum). The current fixed bottom-right collides with the Leaflet zoom control on small viewports.
3. **Collapsible legend** in the runtime (click the header to collapse). Necessary for maps with many categorical layers where the legend can overflow.
4. A legend is **not mandatory** for every layer. Keep the per-layer `legendEnabled` toggle and the project-level `legendShow` switch. Sensible default: layers with categories → legend on; single-symbol layers → legend off.

### 3.6 Add a Preview menu near Import / Export (Pain Point 6)

**Recommendation (matches the user request)**

1. **New "Preview" button** in the topbar between Save Local and Export ZIP. Use the lucide `Eye` icon.
2. **Preview mode** renders the full exported layout (with the inspector and editor sidebar removed) in either a full-screen overlay or a `Blob`-URL iframe. Two implementation choices:
   - **Option A (preferred):** an in-app overlay that hides the inspector and renders `MapCanvas` in `preview = true` mode (no draw toolbar, no attribute table, no editor sidebar). Buttons: "Exit Preview" and "Export Now". Fast, no rebuild required.
   - **Option B:** generate the ZIP in memory, extract `index.html` into a Blob URL, render it in an iframe. More accurate (executes `q2ws-runtime.js`) but slower.
3. **Per-layer preview** in the layer list: an "👁 preview only this layer" button that switches `viewMode` to `selected` and zooms to the layer extent. Some of the plumbing already exists via `viewMode: "selected"` ([src/App.tsx:386](../src/App.tsx#L386)).
4. When the user clicks **Export ZIP**, surface a small confirmation: "Preview before exporting?" with `[Preview]`, `[Export now]`, `[Cancel]`.

## 4. System-Wide Findings (beyond the six pain points)

### 4.1 Data and Web Security

**Status: healthy with a few defense-in-depth opportunities.**

What is already strong:

- The GeoJSON parser is pure `JSON.parse`, never `eval` or `new Function` ([src/lib/qgis2webParser.ts:108](../src/lib/qgis2webParser.ts#L108)).
- Every user-supplied string that reaches HTML goes through `escapeHtml()` (runtime.ts 83-96, MapCanvas.tsx 280-281, 133).
- No `dangerouslySetInnerHTML` anywhere in `src/`.
- `npm audit`: 0 vulnerabilities.
- COOP `same-origin` is set, providing top-window isolation.

Recommended hardening:

1. **Inject a CSP `<meta>` tag** into the exported `index.html` ([src/lib/exportProject.ts:85-94](../src/lib/exportProject.ts#L85)). Minimum:
   ```html
   <meta http-equiv="Content-Security-Policy"
         content="default-src 'self'; img-src 'self' data: https:; style-src 'self' 'unsafe-inline'; script-src 'self'; connect-src 'self' https:;">
   ```
   Tiles are served from external https origins so `img-src https:` is required. `'unsafe-inline'` for styles is unfortunately required by Leaflet (it sets inline styles on its DOM nodes).
2. **Sanitize logo upload.** The logo currently arrives as a data URL (Branding section in [src/App.tsx](../src/App.tsx)) without MIME validation. Restrict to `image/png|jpeg|svg+xml|webp`, cap size at ~512 KB, and run SVGs through DOMPurify with the SVG profile to strip `<script>`.
3. **Validate ZIP entries.** Check for `..` path traversal at unzip time ([src/lib/fileImport.ts](../src/lib/fileImport.ts)). `__MACOSX/` is already filtered (line 30), but path traversal is not.
4. **Cap file count on import.** A typical qgis2web folder has fewer than 500 files. Add a "ZIP contains >5,000 files. Continue?" guard to mitigate ZIP bombs.

### 4.2 Robustness with large datasets

The browser will lock up around 10k features. There are no safeguards today.

Recommendations

1. **Threshold warning at import.** Compute the total feature count during parsing; if it is greater than 10,000, surface a dialog: "Layer X has 45,231 features. Rendering may be slow. Continue?".
2. **Auto-simplify for preview.** Run `@turf/simplify` in a Web Worker for layers with more than 5k features at low zoom; restore full geometry at higher zoom levels.
3. **Marker clustering** for Point layers with more than 500 features. Use `leaflet.markercluster` (mature, lightweight).
4. **Canvas renderer** is already enabled (`preferCanvas: true` at [MapCanvas.tsx:43](../src/components/MapCanvas.tsx#L43)). Keep it.
5. **Pagination + filter in the attribute table.** Required once the 80-row cap is removed.
6. **Worker-based heavy ops.** `studioWorker.ts` already handles parsing. Add workers for: simplification, GeoJSON → ArrayBuffer encoding for OPFS, and bbox computation.
7. **OPFS quota check.** Before writing `last-project.json`, call `navigator.storage.estimate()`; warn the user when `usage / quota > 0.8`.

### 4.3 Error handling and diagnostics

- Runtime config load failure ([runtime.ts:161](../src/runtime/runtime.ts#L161)) is a `console.warn` only. Add a minimal error overlay in the runtime: "Studio runtime failed to initialize. Check `q2ws-config.json`."
- OPFS load failure ([src/lib/opfs.ts:25](../src/lib/opfs.ts#L25)) silently returns null. Surface a toast instead of failing silently.
- Tile errors are unhandled (see 3.3 #3).
- A `diagnostics` array already exists on the project type ([src/types/project.ts:135](../src/types/project.ts#L135)). Surface it in a footer of the side panel as a Diagnostics panel.

### 4.4 Code quality and consistency

- `App.tsx` is 806 lines. Split into: `components/Topbar.tsx`, `components/SidePanel.tsx`, `components/Inspector/{BrandingTab,LayerTab,StyleTab,PopupTab,LegendTab}.tsx`, `components/AttributeTable.tsx`, `components/EmptyState.tsx`.
- `styles.css` is 600+ lines. Break it up per component during the shadcn migration.
- State is `useState`-only inside `App.tsx`. Tolerable today; if complexity grows, consider `zustand` (small, React 18 friendly).
- `LayerStyle` in `project.ts` mixes point, line, and polygon properties. Consider a discriminated union keyed on `symbolType`.
